# Tech Debt Scout

## Overview

This plugin systematically crawls your code repositories to automatically map, prioritize, and surface code rot, deprecated library usages, and complex technical debt hotspots.

- **The Task:** Surfacing tech debt across the organization's repositories.
- **The Logic:** A crew periodically scans repositories for deprecated libraries or "TODO" comments, creating Jira tickets or Backstage Scorecard updates.
- **Framework:** **LangGraph** for the role-based scan (Scanner, Triager, Reporter).

## Dependencies & Mock Targets

This assistant operates a **scheduled multi-agent governance workflow**. It scans code repositories, evaluates the architectural severity of identified anomalies, and routes tracking tickets or compliance dashboard updates to the proper domain teams.

### 1. Core Backstage Services (`coreServices`)

- **`coreServices.urlReader`**: Crucial for the _Scanner_ agent to pull down source files, look for code markers like `TODO` or `FIXME`, and extract package dependency files (`package.json`, `go.mod`).
- **`coreServices.database`**: Manages the persistent execution history, stores deduplication hashes (to prevent creating duplicate tickets for the same code block), and powers LangGraph checkpoints.
- **`coreServices.scheduler`**: Dictates the periodic sweep configurations (e.g., triggering a full repository debt audit every Sunday at midnight).

### 2. Sibling Plugins & Data Sources

- **`CatalogBackend` / `CatalogService`**: Provides the structural target directory. The agent parses the catalog to locate active software assets and find their respective `://github.com` annotations.
- **`Scorecards` / `Soundcheck` Plugins**: The primary internal compliance target. The _Reporter_ agent uses this backend service to automatically update a component's Tech Debt health score or pass factual metrics to code-quality widgets.
- **Jira / Linear Plugins**: The primary outward-facing tracking targets where the agent opens tasks for engineering teams.

## Testing Strategy

Your **LangGraph** framework maps this out as a multi-node graph partitioned into distinct roles: a **Scanner Node**, a **Triager Node**, and a **Reporter Node**. Your testing strategy must verify that the shared graph state cleanly updates as context moves between these specialized agents.

```bash
Scanner Node ──────> Triager Node ───────> Reporter Node

 Reads files &       Filters out           Creates Jira
 extracts debt       low-priority          tickets or up-
 indicators          items / chores        dates Scorecard
```

### 1. Validating Role-Based Data Separation

You need to verify that when the _Scanner_ node passes a raw array of 100 `TODO` strings to the shared state channel, the _Triager_ node correctly uses its LLM policy rules to ignore simple text comments while escalating a high-severity item (e.g., an outdated library version facing an active CVE).

Inject these dependencies into your Backstage backend test harness using custom service factories:

```typescript
import {
  createServiceFactory,
  createServiceRef,
} from '@backstage/backend-plugin-api';
import { startTestBackend, mockServices } from '@backstage/backend-test-utils';
import { techDebtAiScoutPlugin } from '../plugin';

// 1. Mock GitHub file reader to return code containing specific debt flags
const mockGithubCodebaseFactory = createServiceFactory({
  service: createServiceRef<any>({ id: 'github.service' }),
  deps: {},
  async factory() {
    return {
      getFileContent: async (repo: string) => `
        // TODO: temporary hack, fix this before deploying to production
        const processPayment = (amount) => { return true; };
        // FIXME(security): hardcoded encryption salt string
        const salt = "fallback-secret-salt";
      `,
    };
  },
});

// 2. Mock Jira service to catch and assert ticket creations
const mockJiraTrackerFactory = createServiceFactory({
  service: createServiceRef<any>({ id: 'jira.service' }),
  deps: {},
  async factory() {
    return {
      createDebtTicket: async (summary: string, priority: string) => {
        // Assert that the Triager escalated the hardcoded salt but ignored the generic TODO string
        if (
          summary.includes('hardcoded encryption salt') &&
          priority === 'HIGH'
        ) {
          return { ticketId: 'JIRA-901' };
        }
        throw new Error(
          `Unexpected ticket generated by Reporter node: ${summary}`,
        );
      },
    };
  },
});

// 3. Execute the LangGraph role-based validation
describe('Tech Debt Scout LangGraph Execution', () => {
  it('should route text issues through the Scanner, Triager, and Reporter nodes accurately', async () => {
    const { server } = await startTestBackend({
      features: [
        techDebtAiScoutPlugin(),
        mockGithubCodebaseFactory(),
        mockJiraTrackerFactory(),
        mockServices.catalog.factory({
          entities: [
            {
              apiVersion: 'backstage.io/v1alpha1',
              kind: 'Component',
              metadata: {
                name: 'payment-svc',
                annotations: { '://github.com': 'org/payment-svc' },
              },
            },
          ],
        }),
        mockServices.database.factory(), // Preverves deduplication hashes across test runs
        mockServices.rootConfig.factory({ data: {} }),
      ],
    });

    // Step A: Trigger the scout background task using the scheduler test helper.
    // Step B: Assert that the LangGraph processes the code stream through all three role nodes.
    // Step C: Confirm that the final state machine context records only the high-severity ticket reference.
    // Step D: Verify that a second consecutive execution execution does not duplicate the Jira write action.
  });
});
```

### 2. Guarding Against Ticket Duplication Loops

Because this plugin runs continuously on a cron loop, testing **idempotency** is vital. Ensure that your database layer stores a tracking hash of the identified code snippet (e.g., file path + issue line content). Write a test case that runs the graph twice against the exact same mock repository data. Inspect the PostgreSQL `mockServices.database()` instance to assert that the graph blocks duplicate actions at the _Reporter_ boundary.
